# Reconciliation

## Что такое? Какую функцию выполняет?
Это всё о DOM и о том, как медленно в него что-то добавляется или удаляется. Когда мы пишем что-то на React, наша главная цель это позволить ему преобразовать всё, что мы ему даём в обычные DOM-элементы с соответствующими данными.

> **Reconciliation** - процecc, в котором React сравнивает предыдущее виртуальное дом дерево с новым и решает, что нужно обновить в реальном DOM

> React сравнивает дееревья по: **тип элемента**, **позиция в дереве** и **key**.

1. при новом рендере строит новое описание UI из JSX;<br/>
Каждый раз, когда меняется state, props или context, React повторно вызывает компонент.
```ts
{ type, props, key }
```
2. имеет предыдущее дерево, связанное с прошлым рендером;<br/>
В React уже есть результат прошлого рендера.
React сопоставляет прошлое и новое дерево через Fiber-структуры и вычисляет, как обновить реальный DOM.

3. **Во время reconciliation сопоставляет старые и новые элементы**
- По типу
```js
// По типу
<div />
<span />
```
- По позиции элемента в дереве
> Если элемент того же типа стоит на том же месте, React обычно пытается переиспользовать существующий instance/DOM node.
- По `key`
> Без `key` React ориентируется по позиции, С `key` - по идентичности.

```md
┌──────────────────────────────┐
│ 1. Что-то вызвало update     │
│    - setState                │
```

>  React Reconciliation - это механизм сравнения предыдущего и нового дерева React-элементов после рендера. 
Его задача - определить минимальный набор изменений, который нужно применить к UI. React не сравнивает DOM напрямую на каждом шаге, а работает через virtual tree и эвристики: если тип элемента меняется, он считает узел новым и размонтирует старый subtree; 
если тип тот же, пытается переиспользовать существующий узел и сохранить `state`.
 Для списков ключевую роль играет `key`, потому что он задаёт identity элемента. 
 Если `key` выбран неправильно, особенно index в динамическом списке, можно получить баги с переносом `state` и некорректным обновлением input’ов. 
 На практике reconciliation нужен для производительности и корректного сохранения состояния компонентов.


## Type в Virtual DOM
> Virtual DOM это огромный JavaScript объект со всеми компонентами которые нужно отрисовать, все их свойства и дочерние компоненты (children), которые по своей сути являются такими же объектами. Обычное дерево.

```JSX
const Input = ({ id, label }) => {
  return (
    <React.Fragment>
      <label htmlFor={ id }>{ label }</label>
      <input type="text" id={ id } />
    </React.Fragment>
  )
}
```

Virtual Dom будет выглядеть так
```js
[
  {
    type: 'label',
    ... // другие свойства
  },
  {
    type: 'input',
    ... // другие свойства
  }
]
```
DOM элементы вроде input и label имеют свой тип, представленный строкой, и React знает как напрямую преобразовывать это в DOM. Но компоненты не связаны напрямую с DOM элементами, и поэтому React должен как-то это обойти.

```jsx
const Component = () => {
  return <Input />
}
```
>!!! В таком случае в качестве `type` используется **ссылка на функцию**. React просто берёт функцию, известную как Input и использует её в качестве значения поля `type`.
```js
{
  type: Input, // ссылка на функцию Input, которую мы объявили ранее
  ... // другие свойства
}
```
Когда React получает команду на рендер интерфейса, он итерационно проходит по всему дереву со следующими инструкциями:
- *если type является строкой, то генерируем указанный в строке HTML-элемент.*
- *если type это функция (как в примере выше), то вызываем её и проходимся по дереву, которое эта функция нам вернула.*
> И так до тех пор, пока не получится дерево DOM элементов готовых к рендеру
```js
const Component = () => {
  return (
    <div>
       <Input placeholder="Text1" id="1" />
       <Input placeholder="Text2" id="2" />
    </div>
  )
}
```
будет представлен как:
```js
{
  type: 'div',
  props: {
    // children are props!
    children: [
      {
        type: Input,
        props: { id: "1", placeholder: "Text1" }
      },
      {
        type: Input,
        props: { id: "2", placeholder: "Text2" }
      }
    ]
  }
}
```
> Ну и наконец, когда всё дерево будет готово, React добавит Virtual DOM элементы в актуальный document используя `appendChild`

При изменении стейта или пропсов:

> React сравнит поле `type` этого объекта до и после обновления стейта. Если `type` такой же, то компонент Input будет помечен "needs update" и запустится его обновление. Если `type` изменился, тогда React в процессе цикла ре-рендеров удалит (размонтирует) старый компонент и добавит (смонтирует) новый. 

Пример:
```js
const Component = () => {

  if (isCompany) return <Input />

  return <TextPlaceholder />
}
```
и предположим, что обновление было запущено из-за смены значения isCompany с true на false, то объекты, которые сравнивает React будут такими:

```js
// до обновления isCompany равно "true"
{
  type: Input,
  ...
}

// после обновления, isCompany уже "false"
{
  type: TextPlaceholder,
  ...
}
```
